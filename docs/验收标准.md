
# 扩梦验收标准

## 1. 功能验收标准

### 核心功能

- **JSON 格式与 IR 生成**：是否符合规范，能正确解析与生成 IR。
- **Action 执行、状态管理、动画控制和事件调度**：是否按预期工作，状态变化是否及时渲染。
- **动态 Bundle 加载与解析**：是否能够在各平台成功加载、解析并运行。
- **前台渲染效率**：是否能避免过多的重绘和布局计算，保持高效渲染。
- **平台支持**：各平台（iOS, Android, Web）核心功能是否正常运行，特别是 Clipboard、网络请求、文件读写等。

### 平台适配

- **移动端适配**：iOS 和 Android 平台的运行情况，包括状态变化与 UI 更新。
- **Web 适配**：Web 端能否正确通过 WASM 或适配器支持动态更新与渲染。
- **小程序平台适配**：QORM的跨平台设计也需要支持主流小程序平台（如微信小程序、支付宝小程序等）。对于小程序的适配，考虑到其特殊的限制和性能要求，QORM应进行如下优化：
1. **平台适配器与插件支持**
    * 适配器：为不同小程序平台（如微信小程序、支付宝小程序等）提供专门的适配器，确保QORM的渲染、事件处理和资源加载机制能够兼容各个平台的特点。
    * 平台特定API封装：小程序平台有不同的API接口，如事件系统、UI组件等。QORM可以提供统一的API封装，将平台特有的API与QORM的核心渲染系统进行整合。
2. **资源管理与加载**
    * 资源加载优化：小程序平台对包体积有严格限制，QORM需要优化资源加载机制，确保动态加载资源时不超过平台限制，且加载速度迅速。
    * 缓存与懒加载：支持小程序的缓存机制，并通过懒加载策略按需加载资源，避免一次性加载过多资源导致性能瓶颈。
3. **UI组件与渲染适配**
    * UI组件库适配：小程序有自己的UI组件库（例如微信小程序的WXML、WXSS），QORM应提供适配层，将QORM的UI组件映射到小程序的组件上。为此，可以通过小程序自定义组件的方式提供一致的UI体验。
    * 渲染管线优化：在小程序平台上，渲染效率和内存管理尤为重要。QORM的渲染管线需要进行优化，减少重绘和布局计算，特别是在低性能设备上，以确保UI更新的流畅性。
4. **性能优化**
    * 渲染性能：确保在小程序环境中，UI的渲染能够保持高效。特别是在低性能设备上，渲染效率需要经过精细调优，以避免卡顿和延迟。
    * 内存管理：小程序对内存的使用有严格要求，QORM需要严格监控内存使用，避免内存泄漏，确保在长时间运行后，内存占用不超过平台限制。
5. **调试与测试**
    * 调试工具支持：提供适合小程序的调试工具支持，例如通过小程序的开发者工具进行UI渲染的调试，帮助开发者快速定位问题。
    * 测试用例：为小程序平台添加专门的测试案例，涵盖资源加载、UI渲染、事件响应等方面的测试，确保小程序在不同设备上的稳定性和流畅性。

## 2. 性能验收标准

### 运行时性能

- **加载时间**：首次加载、缓存加载的时间是否满足预定标准。
- **UI 响应**：状态更新、动画执行时是否达到 60 FPS。
- **渲染平稳性**：帧率是否保持稳定，UI 渲染不会卡顿。

### 内存使用

- **内存泄漏检测**：是否有过高的内存使用，内存是否有泄漏。
- **性能基准对比**：与同类应用的内存使用对比，是否在合理范围内。

### 渲染性能

- **不同设备渲染效率**：在低性能设备上的渲染效率是否合格。
- **动画和文本渲染**：是否能在不同分辨率下快速响应，保持流畅的动画。

## 3. 安全验收标准

### 权限控制

- **动态更新权限管理**：是否能严格控制权限，避免非授权的权限修改。
- **安全问题**：是否存在绕过权限控制的问题。

### 数据保护

- **Bundle 和 Patch 安全性**：是否对 Bundle 和 Patch 文件进行了有效签名与加密，防止篡改。
- **数据隐私**：是否遵循数据保护法规，防止用户数据泄露。

### 合规性

- **平台合规**：是否符合 Apple 和 Google 的动态代码执行要求（App Store、Play Store 发布合规）。
- **隐私保护**：是否符合各平台的隐私保护政策。

## 4. 文档验收标准

### 文档完整性

- **核心文档**：是否包含 README、架构图、开发指南等文档，内容是否完整且易于理解。
- **平台适配文档**：是否明确说明了各平台的适配策略及注意事项。

### 开发文档

- **开发流程文档**：是否包含详细的开发指南、模块结构、API 说明等。
- **测试策略**：是否有明确的测试策略与验收标准。

### 测试文档

- **测试标准**：是否有详尽的测试文档，包含功能测试、性能测试、集成测试等。
- **自动化测试**：是否涵盖自动化测试与 CI/CD 流程。

## 5. 集成与部署验收标准

### CI/CD

- **自动化构建与测试**：是否有完善的 CI/CD 流程，能够自动完成构建与测试。
- **部署流程**：是否能够顺利部署到各平台，并保持稳定性与性能。

### 打包与发布

- **发布包完整性**：发布包是否包含完整的代码、资源文件、文档及测试用例。
- **发布验证**：是否能在目标平台上成功安装并正常运行。


## 6. 潜在风险与设计审查验收标准

### 表达式引擎与响应式系统

- **表达式计算边界**：是否明确规定了表达式引擎的支持子集（如是否图灵完备，抑或仅支持基础算术与逻辑运算），并在实现上避免了不必要的 WASM 体积膨胀或执行性能开销。
- **深层状态依赖（Dirty Tracking）**：针对复杂数据结构（如嵌套对象属性的修改、数组的增删操作），是否具备精准的脏数据追踪机制，能够正确且高效地触发局部布局重算与渲染，严禁无谓的全场景重绘。

### Web 端渲染与无障碍访问 (a11y)

- **首屏加载与体积控制 (Performance Budget)**：在基于 Go 服务端渲染（输出 HTML/CSS）与离线 Go→WASM（`cmd/qorm-wasm`）客户端的架构下，运行时体积（尤其 WASM 包）与依赖是否被控制在合理的首屏启动性能阈值内。
- **无障碍访问与 SEO 映射**：QORM 渲染为真实 HTML/CSS DOM，需确保语义化元素、无障碍属性（Accessibility Tree）到位，以保证对屏幕阅读器的兼容性及基础 SEO 的支持。

### Agent Patch 的健壮性

- **Patch 路径定位鲁棒性**：在 Agent 预览、生成并应用 Patch 时，是否优先采用了基于稳定语义 ID（如 `find node by id`）的修改策略，以防在 UI 层级或数组顺序发生细微变化时，基于传统绝对路径或索引的 JSON Path 产生失效或致命性修改冲突。

### 组件系统完整性

- **复杂组件通信规范**：组件模型的设计是否完备，是否在规范与实现中清晰定义了 Props 的流转传递机制、内外部事件的冒泡与回调（Event Emitting）规则，以及插槽（Slots）/ 子节点内容的动态注入与分发逻辑。

---

**验收标准说明**

本验收标准适用于 QORM 项目的各个平台和版本。验收的核心目的是确保 QORM 在各个环境下的核心功能、安全性、性能、文档和部署都达到规定要求，并在产品发布前完成全面测试。
